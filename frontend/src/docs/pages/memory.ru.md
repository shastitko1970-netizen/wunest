# Память чата

В длинных чатах модель забывает начало. WuNest даёт три параллельных механизма «памяти». Можно использовать любой из них, можно комбинировать.

Все три живут в **drawer чата (иконка `mdi-tune-variant` в шапке) → вкладка Memory**.

| Механизм | Как срабатывает | Что попадает в промпт |
|---|---|---|
| **Author's Note** | Безусловно, на каждом запросе | Свободный текст, в позицию которую ты выбрал |
| **Авто-саммари** | Когда в prompt уходит ≥ `threshold` токенов | Сжатый recap старой части истории |
| **Заметки и pinned facts** | Безусловно, на каждом запросе | Список ручных «не забывай о Х» в системном блоке `[Memory]` |

---

## Author's Note

Свободный текст, **подмешивается в каждый запрос**. Главный инструмент жанровой направляющей и быстрых правок «модель опять делает Х, скажи ей не делать».

В WuNest (после M53) Author's Note редактируется в **двух** местах:

- **Memory tab** — основной редактор (M53 — добавлен ради discoverability)
- **Generation tab** — тот же редактор, для удобства когда уже там

Оба синхронизированы. Источник истины — `chat_metadata.authors_note`.

### Поля

| Поле | Что значит |
|---|---|
| **Текст** | Содержимое заметки. Markdown работает. Macros (`{{user}}`, `{{char}}`, `{{time}}`, …) раскрываются. |
| **Глубина** (`depth`) | На сколько сообщений с конца истории вставить. `0` — после последнего user-сообщения. `1` — за одно сообщение до конца. И т.д. |
| **Роль** (`role`) | `system` (дефолт), `user`, или `assistant`. Контролирует «откуда» модель воспринимает заметку. |

### Куда что вставляется

При `depth = 0` Author's Note вставляется **в самый конец истории, прямо перед запросом ответа**. Это самая «громкая» позиция — модель видит note последним.

При `depth = 1, 2, …` — вставляется на N сообщений раньше. Полезно для «постоянного контекста, но не самого последнего слова».

### Когда какую роль использовать

- **`system`** (дефолт) — для жёстких правил и инструкций. «Это нуар. Не ломай четвёртую стену.»
- **`assistant`** — для in-character инструкций. «Помни, что у Алисы сломана левая нога.» Звучит как «инсайт самого персонажа».
- **`user`** — реже. Для «фейкового» user-напоминания «Кстати, у меня нет денег» — модель примет как канон.

### Что туда писать

- **Жёсткие ограничения сцены** — «Это мрачный нуар, никаких jokes», «Никогда не выходи из роли»
- **Long-running facts** не помещающиеся в карточку — «У Alice сломана левая нога, она использует трость»
- **Стиль письма** — «Отвечай в стиле Hemingway: короткие предложения, без украшений»
- **Жанр и тон** — «Это RPG в духе Disco Elysium: внутренние мысли = курсив»

### Что НЕ писать

- **Меняющиеся факты** — будешь править note каждый ход. Лучше через [лорбук](lorebooks).
- **Большие тексты** — это идёт в **каждый** запрос. Бюджет токенов уходит в воздух.
- **Биографию персонажа** — это в карточке.

### Token budget

Author's Note 200–400 символов = ~50–100 токенов. На каждом запросе. На длинном чате — десятки тысяч токенов суммарно. Поэтому держи лаконично.

## Авто-саммари (rolling summary)

**Memory tab → Auto-summary**.

Toggle ON → когда в prompt уходит ≥ `threshold_tokens` токенов, фоном запускается LLM-вызов, который сворачивает старую часть истории в **сводку** и подмешивает её в системный промпт.

### Как это работает технически

1. После каждого успешного ответа модели измеряется `tokens_in` (сколько ушло в **input** последнего запроса).
2. Если `tokens_in >= threshold_tokens` — фоновая горутина вызывает указанную модель саммари с шаблоном `[Previous summary] + [recent messages] → updated summary`.
3. Сводка сохраняется в БД с ролью `auto`. Перезаписывает предыдущую.
4. На следующем запросе саммари клеится в системный блок `[Memory] / ## Rolling summary of earlier events`.

### Параметры

| Поле | Что |
|---|---|
| **Enabled** | On/off |
| **Threshold (tokens)** | На каком объёме `tokens_in` запускать. Дефолт ~2000–4000. Слишком низко → каждый ход. Слишком высоко → саммари редко обновляется. |
| **Model** | Модель саммари — отдельная от твоей основной. Можно дешевле (Gemini 2.5 Flash, Haiku, gpt-4o-mini). |
| **BYOK** | Если хочешь использовать свой ключ для саммари (не WuApi pool). |

### Per-chat мутекс

Один фоновый саммари на чат за раз. Если триггер сработал во время уже идущего саммари — пропускается (уже идущий «догонит»).

### Retry-and-silent-fail

Если саммари упало (rate limit, network) — backoff 5s → одна попытка retry → fail-silent в slog. Юзера не спамим toast'ом «не удалось саммаризировать», feature не отключается автоматически.

### Когда полезно

- Чат >50 ходов на 8K-context модели — без саммари модель забывает начало.
- Долгий ролеплей где важно «помни наш сговор в 5-м ходе».
- Хочешь сэкономить — пусть дешёвая Haiku сжимает, основная Sonnet работает на сжатой истории.

### Когда не нужно

- Чаты до 30 ходов на 1M-context моделях — окно ещё не упирается.
- Один-двухходовые сессии (пробовать карточку).
- Если важна **дословная** память каждого хода — саммари по определению lossy.

## Заметки и закреплённые факты

**Memory tab → Notes**.

Свободные заметки, которые ты пишешь сам и которые **всегда** попадают в промпт. Не сжимаются авто-саммаризатором.

### Два типа

| Роль | Где в промпте | Когда |
|---|---|---|
| **Pinned** (закреплённые) | В блоке `[Memory] / ## Key facts (always-on)` — самое «громкое» место | Канон, который модель не должна забывать |
| **Manual** (обычные) | В блоке `[Memory] / ## Notes` — после саммари | Полу-каноничные заметки, эпизодные напоминания |

Обе роли применяются **на каждом запросе**, без условий. В отличие от лорбука, где запись срабатывает по ключу.

### Когда какую

- **Pinned** — «У Морган только одна нога», «Действие на Марсе», «Дочь Алисы зовут Лина». Жизненный канон.
- **Manual** — «Они недавно поссорились», «Идёт дождь третий день». Текущие обстоятельства, которые могут смениться.

### Где это в UI

В Memory tab:
1. **Pinned** — видны в отдельной плашке (со звёздочкой)
2. **Manual** — обычный список ниже
3. **Add** — новый input с переключателем «обычная / закреплённая»

Можно редактировать, перепинить (звёздочка), удалить. Pinned всегда выше manual в собранном промпте.

### Связь с авто-саммари

Когда срабатывает авто-саммари — он **видит** твои manual/pinned (они тоже идут в шаблон саммари). Но саммари их **не переписывает**: твои заметки остаются как есть, авто-сводка живёт параллельно.

Если ты редактируешь автосгенерённую сводку — её роль автоматически меняется на `manual`, чтобы следующий проход auto-summary её не перезатёр.

## Что выбирать когда

| Задача | Используй |
|---|---|
| «Не выходи из роли» | Author's Note (`system`, depth 0) |
| «Помни этот канон навсегда» | Pinned note |
| «У Алисы кошка Мурка, может всплыть» | Лорбук с ключом `Мурка, кошка` |
| «У персонажа всегда тёмный плащ» | Карточка → description |
| «Чат стал длинным, режу токены» | Авто-саммари + дешёвая модель саммари |
| «Сейчас идёт дождь» | Manual note (когда погода сменится — отредактируешь) |
| «Случайно генерим бросок d20» | Macros в карточке (`{{roll::d20}}`) |

## Расход токенов

| Механизм | Типичный размер | Токены/turn |
|---|---|---|
| Author's Note (200 chars) | small | ~50 |
| Авто-саммари (1K chars) | medium | ~250 |
| Pinned facts (5×100 chars) | small | ~125 |
| Manual notes (10×100 chars) | medium | ~250 |

Все четыре в сумме обычно дают **+500–1000 токенов** на каждом запросе. На 1M-context моделях незаметно. На 8K (старый GPT-4, Claude Haiku 1) — съедает ощутимо.
