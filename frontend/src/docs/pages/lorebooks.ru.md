# Лорбуки (World Info)

Лорбук — это **условный канон**. Вместо того чтобы пихать всю вселенную в карточку (где она съедает токены каждый ход), ты раскладываешь её на **записи**, которые подгружаются только когда упомянуты в диалоге.

Если карточка персонажа отвечает на вопрос «**кто** перед тобой», лорбук отвечает на вопрос «**что** есть в этом мире и **когда** оно становится важно».

---

## Как это работает

На каждом запросе активатор:

1. Сканирует последние N сообщений (см. **глубина**).
2. Для каждой записи в подключённых лорбуках проверяет: попадает ли её **ключ** в окно сканирования.
3. Если да — добавляет содержимое записи в системный промпт.
4. Делает до **3 рекурсивных проходов** — содержимое сработавших записей расширяет окно сканирования (если запись не помечена `exclude_recursion`).

В итоге модель видит карточку + сработавшие записи + историю сообщений.

---

## Когда писать в лорбук, а не в карточку

| Кладёшь в карточку | Кладёшь в лорбук |
|---|---|
| Имя, внешность, голос | Места, через которые персонаж изредка проходит |
| Главные отношения (близкие) | Второстепенные NPC, имена |
| Привычки и манеры | Предметы / артефакты (только когда упомянуты) |
| Каноничный backstory (1-2 факта) | Подробная история мира, фракции, события |
| Что важно знать **всегда** | Что важно знать **иногда** |

Эмпирика: если запись срабатывает реже чем в 1 ходе из 5, она уместнее в лорбуке.

---

## Создание

**Библиотека → Лорбуки → Новая** или **Импорт** для JSON-файлов SillyTavern. Поддерживаются и массивная (`entries: [...]`), и объектная (`entries: { "0": {}, "1": {} }`) форма.

После создания лорбук появляется в карточке персонажа: **Library → Characters → карта → иконка «Лорбуки»** → multi-select. Только привязанные книги активируются для этого персонажа.

## Запись: поля и смыслы

### Ключи (`keys`)

OR-матчинг по списку строк. Любое попадание в окно сканирования активирует запись.

#### Как писать ключи

- **Ключ — это лексема, не идея.** «Король» не сработает на «королевство». Если важно — добавь оба ключа.
- **Включай вариации.** `Алиса, Алисы, Алисе, Алису` — модель называет персонажа по-разному.
- **Не дроби лишнее.** Если 90% случаев модель пишет «Морган», достаточно ключа `Морган`.
- **Избегай слишком общих.** `он`, `время`, `сказал` сработают почти всегда → запись будет triggering на каждом ходу = нет смысла иметь её в лорбуке.

#### Целые слова (`match_whole_words`)

`cat` без флага сработает на `concatenate`. С флагом — нет. Оборотная сторона — `cat` тогда не сработает на `cats`. Включай если ключ короткий и часто встречается как подстрока в нерелевантных словах.

### Содержимое (`content`)

Текст, который вставляется в системный промпт когда запись активируется.

#### Стиль

- **Пиши третьим лицом.** «Морган — кузнец…», не «я кузнец».
- **Конкретика.** «Использует молот с рукоятью красного дерева» лучше, чем «классный кузнец».
- **Краткость.** 50–250 слов на запись. Если больше — раздели на две.
- **Без избыточной сюжетки.** Лорбук — это справочник, не глава романа. Глава — в чате.
- **Markdown работает.** Заголовки `## Внешность`, списки.

#### Длина

Разумный ориентир — суммарно сработавшие записи не должны превышать 30% длины карточки. Иначе они начинают «перебивать» персонажа.

### Selective + Secondary keys

Если включён флаг **«Требует вторичный ключ»** (`selective`), для активации нужны **И** primary, **И** secondary. Полезно когда основной ключ слишком общий:

- Primary: `Морган`
- Secondary: `прошлое, история, юность, родители`

Запись «backstory of Morgan» сработает только когда обсуждается **именно** прошлое Моргана — а не каждое упоминание имени.

### Позиция (`position`)

Куда вставится содержимое сработавшей записи в финальном промпте:

| Позиция | Куда |
|---|---|
| `before_char` | Перед описанием персонажа (дефолт). Хорошо для мира / контекста. |
| `after_char` | После описания. Для уточнений к персонажу. |
| `at_depth N` | На глубину N сообщений с конца. Для "напоминаний у самого хода". |
| `before_an` / `after_an` | Перед / после Author's Note. Для тонкой подстройки приоритетов. |

Все четыре позиции **работают в v1**. Если нужно «всегда напомнить вот это прямо перед моделью думает», ставь `at_depth 1`.

### Глубина (`depth`)

Сколько последних сообщений сканировать на ключи. `0` — глобальный дефолт книги (обычно 4). Чем больше — тем длиннее «память ключей», но и тем чаще ложные срабатывания.

### Порядок вставки (`insertion_order`)

Меньше = раньше в промпте. При тай-брейке: индекс книги → индекс записи. Используй когда у тебя несколько записей одной темы и важна последовательность.

### Вероятность (`probability`)

Случайный гейт от 0 до 100.

- `0` или `100` — всегда срабатывает (если ключ совпал)
- `1`–`99` — random roll: с вероятностью N% запись активируется

Полезно для вариативности (например, разные «эффекты погоды»).

### Группа (`group`)

Имя группы (произвольная строка). В пределах одной группы за проход активируется **максимум одна** запись (с самым низким `insertion_order`).

#### Когда нужно

- Несколько вариантов «приветствия таверны» — одна срабатывает, другие подавляются.
- Альтернативные описания одного и того же объекта.
- Эффекты погоды / времени дня.

#### Обходить групповое ограничение (`group_override`)

Флаг на записи. Если включён — запись активируется независимо от группы (как будто группы нет).

### Постоянная активация (`constant`)

Запись срабатывает **на каждом ходу**, без проверки ключей. Используй для «всегда напоминай о Х». По эффекту похоже на Author's Note, но привязано к лорбуку → переносится с персонажем.

### Рекурсия

Активатор делает до **3 рекурсивных проходов**: после первого матча содержимое сработавших записей попадает в окно сканирования, чтобы могли сработать связанные.

- **Не триггерить рекурсию** (`exclude_recursion`) — содержимое **этой** записи НЕ попадает в окно следующего прохода. Используй для «описательного» текста, где упомянутые имена не должны цеплять другие записи.
- **Не активироваться рекурсией** (`prevent_recursion`) — **эта** запись активируется только на первичном скане истории, не из-за другой записи. Защита от каскада.

### Регистрозависимость (`case_sensitive`)

`Алиса` ≠ `алиса` если флаг включён. По умолчанию — выключен.

## Продвинутые поля (не применяются в v1)

В редакторе записи есть свёрнутая секция «Продвинутые». Поля **сохраняются** с записью (в JSON-экспорте они есть), но **активатор v1 их игнорирует**:

- **Sticky** (ходов) — «держать активной N ходов после первой активации»
- **Cooldown** (ходов) — «минимум N ходов между активациями»
- **Delay** (ходов) — «не активировать в первые N ходов»

Эти три поля требуют сохранения состояния между ходами (per-entry counters), чего v1 не делает. UI помечает эту секцию badge'ом «не применяется в v1». Реализация — в roadmap.

## Импорт из ST

JSON-формат ST совместим. Поддерживаются и массивные, и объектные `entries`. Поля, которые ST умеет, а WuNest пока нет (book-level `recursive_scan` toggle, например), пропускаются без ошибок.

При импорте PNG-карточки её встроенный `data.character_book` автоматически становится отдельным лорбуком и привязывается к персонажу.

## Pattern: типичные раскладки

### Мир-справочник

Один лорбук, ~20 записей: фракции, города, артефакты. Привязывается ко всем персонажам этой вселенной.

### Персонаж-портфель

Один лорбук на персонажа (даже если у тебя одна карточка). 5–10 записей: backstory, второстепенные NPC, локации связанные именно с ним. Карточка — про «здесь и сейчас», лорбук — про «всё остальное».

### Игровая механика

Записи активируются по ключам типа `бросок, urok, проверка, инициатива`. Содержимое — правила. Полезно для «Я хочу D&D-чат, но не хочу пихать правила в карточку».
